# 11. Технический контракт передачи канона (packaging для вендоринга)

> Дополнение к ТЗ «переделка лендинга под бьюти-мастеров». Описывает, **в каком
> виде Claude Design отдаёт результат**, чтобы он без ручной доводки вошёл в
> прод через наш вендоринг-пайплайн. Дизайн-часть — в разделах 1–10; этот раздел
> — только про упаковку и границы ответственности. Текущая вендоренная версия —
> **canon 0.11.0**; целевая поставка — **0.12.0**.

## 11.1 Разделение ответственности (главное — прочитать до макета)

Прод потребляет `@samosite/canon` **напрямую из исходников** (workspace-зависимость
`file:../packages/canon`), поэтому drift между макетом и продом невозможен —
но только если поставка ложится в нашу модель. Модель такая:

| Слой | Кто владеет | Что это значит для этого ТЗ |
|---|---|---|
| **Презентационные компоненты + запечённый демо-контент** | **Канон** (`packages/canon/src/**`) | Карточки-примеры, тарифы, понедельничные кейсы, шаги формы, тексты-рыба демо — живут в каноне. Бьюти-рефокус (примеры, кейсы, 3-карточная витрина тарифов, объекции) поставляется **как изменения canon-компонентов**. |
| **Композиция + порядок секций** | **Консьюмер** (`landing/app/page.tsx` через `<ResponsiveCanonSection>`) | «Стало 8 блоков вместо 13» и порядок блоков — это **наша** сборка: мы монтируем нужные canon-секции в нужном порядке. Удаление секции = мы её не монтируем (компонент может остаться в пакете). |
| **Интерактив, сеть, стейт, аналитика** | **Консьюмер** (`landing/components/*`, `landing/lib/*`) | Поле Hero, стейт-машина модалки, поиск по названию + поллинг черновика, цели Метрики — **наши**. Канон отдаёт презентационные **controlled**-компоненты, а не самодостаточные виджеты. |

Практический вывод для §3–5 ТЗ: **не присылайте самодостаточных виджетов с
собственным fetch/стейтом данных.** Присылайте: (а) новые/переверстанные
презентационные компоненты секций и (б) новые/изменённые controlled-компоненты
шагов формы. Что монтировать и как связать с бэкендом — доделаем на консьюмере.

## 11.2 Формат поставки

- **Zip той же формы, что прошлые:** `export/canon-0.12.0-pkg/` с `src/`,
  `package.json`, `CHANGELOG.md`. `dist/` можно вложить, но он **не источник
  правды** — мы пересобираем его сами (`npm run build`, tsup). Источник правды —
  `src/`.
- **`package.json`:** поднять `version` до `0.12.0` (уникальный номер, не
  переиздавать под старым; если это ре-issue — явно пометить) и обновить
  `description` под содержание релиза. Поле `exports` не ломать — точки входа
  фиксированы: `.`, `./tokens`, `./primitives`, `./landing`, `./intake`,
  `./customer`, `./admin-demo`, `./admin-core`, `./admin-ops`, `./auth`,
  `./source`, `./presets`, `./tailwind-preset`, `./styles.css`.
- **`src/` собирается tsup'ом as-is:** только TSX/TS, React 19 + уже заявленные
  peer-deps (radix, lucide, recharts). **Никаких новых рантайм-зависимостей**
  без явного флага в CHANGELOG (у нас правило «новая зависимость → ADR»).
- **Наш вендоринг-шаг** (чтобы вы понимали, что произойдёт с поставкой):
  `cp` ваших `src/*` → `packages/canon/src/`, бамп `package.json`, `npm run build`
  регенерит `dist/`, `npm install --install-links` в landing, прогон гейтов. Всё,
  что не собирается tsup'ом или ломает существующий импорт, — блокирует вендоринг.

## 11.3 Additive и «не ломать» (drift = 0)

- Все существующие экспорты и пропсы **0.11.0 должны продолжать работать**.
  Изменения — **только additive**: новые пропсы опциональны, дефолт = поведение
  0.11.0. Консьюмер, не передающий новых пропсов, обязан получить прежний рендер.
- Если правка **breaking** (переименован экспорт, изменена обязательная сигнатура)
  — это допустимо, но **явно** пометить `BREAKING` в CHANGELOG + дать migration
  (что именно поменять в вызове). По умолчанию — избегать.
- **Не трогать** консьюмер-файлы (`landing/**`, `backend/**`), `tsup.config.ts`,
  `tokens.ts` без нужды. Правки только внутри `packages/canon/src/`.

## 11.4 Controlled API (обязательно для всего интерактивного)

Паттерн зафиксирован (0.3.0/0.9.1/0.10/0.11):

- Данные — **внутрь пропсами**, действия — **наружу колбэками**. Внутри канона
  **нет** `fetch`/сети, нет доменного стейта, нет `Date.now()`/`Math.random()`.
- **Zero-prop рендерит canvas-mock** (компонент без пропсов показывает демо-состояние
  для артборда). Так интерактивные компоненты остаются проверяемыми в канвасе и
  подключаемыми у нас.
- Экспортировать **TS-типы пропсов** (`FooProps`) для каждого нового
  интерактивного/секционного компонента.

## 11.5 Токены и дизайн-система

- Цвета/шрифты/радиусы/тени — **только из `@samosite/canon/tokens`** (`src/tokens.ts`).
  Хардкод hex в компонентах запрещён. Палитра (уголь/сливочный/оранжевый),
  дисплей-антиква с курсивными акцентами, ритм тёмных/светлых секций — **не менять**
  (раздел 8 ТЗ). Тёмная и светлая темы — обе (консьюмер стемпит `data-theme`).
- Нужен новый токен (например, оттенок под новую бьюти-карточку) — **добавить в
  `tokens.ts` + задокументировать в CHANGELOG**, не хардкодить.

## 11.6 Пресеты и instant-preview под бьюти-подниши (§4.2 ТЗ)

Карточки-примеры и превью в форме рендерятся через `@samosite/canon/presets`
(`PresetRenderer`, семейства × темы; типы `FamilyKey`/`Theme`/`Slot*`). Чтобы
«выбрал брови → видит сайт бровиста» (а не общий пример), нужно **в
`src/intake/rev2.tsx`**:

- **`NICHE_LIB`** — порядок = приоритет: бьюти-подниши первыми (маникюр,
  брови/ресницы, барбершоп/парикмахер, косметолог, + колорист), остальные —
  сгруппировать под «Другое» (флаг группы на элементе, не удаление).
- **`NICHE_DEMO_DRAFTS`** — запись `PreviewDraft` **на каждую бьюти-поднишу** с
  валидными `theme_id`/`family_id` из реальных пресетов (переиспользовать палитры
  карточек из блока «Примеры»). Проверить, что `demoDraftFor(nicheId)` отдаёт
  подниша-специфичный превью.
- Недостающие бьюти-пресеты (ресницы, косметолог/колорист) — добавить в пресеты
  как новые темы/семейства, а не подменять существующие.

## 11.7 DTS и ambient-шимы (почему важен точный CHANGELOG)

Канон собирается с **`dts: false`** (намеренно, `tsup.config.ts`), поэтому типы
экспортов консьюмер держит **вручную** в `landing/types/samosite-canon.d.ts`.
Следствие для поставки:

- В CHANGELOG **явно перечислить каждый новый/изменённый экспорт с точной
  TS-сигнатурой** (имя, пропсы, типы). Тогда расширение шимов на нашей стороне —
  механическое. Пропсовые типы — экспортировать из модуля (`export interface FooProps`).

## 11.8 Аналитические хуки `data-*` (§7 ТЗ — цели Метрики)

Цели вешаются на `data-*`-атрибуты **на стороне консьюмера, без правок канона**.
Значит канон должен **проставить и сохранить стабильные хуки** на новых/изменённых
блоках (менять их потом — только согласованно):

- сохранить: `data-section="<id>"` на каждой секции (нужен и для scroll-целей, и
  для нашего pixel-diff), `data-cta` на primary-CTA, `data-intake-step`,
  `data-niche-id`, `data-candidate-idx`;
- добавить под §7: `data-pricing-tier="<id>"` на карточках тарифов (клики),
  `data-faq-item` на аккордеонах (faq_open — сейчас цель сломана, переведём на
  JS-событие по этому хуку), `data-example-idx` на карточках/слайдах примеров
  (examples_view/клик). Точные имена — согласуем в CHANGELOG.
- `Btn` уже пробрасывает `data-*` на `<button>` (с 0.11) — переиспользовать.

## 11.9 Responsive и visual-regression

- Секции монтируются через `<ResponsiveCanonSection>`: **оба варианта (mobile 390
  + desktop) в одном компоненте**, CSS media-query выбирает нужный. Компоненты —
  либо внутренне-адаптивны, либо экспонируют оба варианта. Mobile-first 390 —
  ведущий (раздел 2 ТЗ).
- Мы pixel-diff'им на 1440/768/390/375. Структура секции должна быть стабильна,
  `data-section` на месте; новые baseline'ы регенерим на своей стороне. Крупные
  необъявленные сдвиги вёрстки в неизменных секциях = красный visual-regression.

## 11.10 CHANGELOG — обязательная верхняя секция 0.12.0

Как в секциях 0.10.0/0.11.0 (у вас перед глазами). Минимум:

1. Одно-абзацное «зачем» + `MINOR, additive` (или `BREAKING` с миграцией).
2. **Новые экспорты** `@samosite/canon/<entry>` — список с TS-сигнатурами пропсов.
3. **Изменённые компоненты** — что за пропсы добавились (additive), новые варианты
   (например `variant='beauty'` у Examples, `PricingCards` рядом с `PricingMatrix`).
4. **`NICHE_LIB`/`NICHE_DEMO_DRAFTS`/пресеты** — что добавлено под бьюти.
5. **`data-*` хуки** — какие проставлены/сохранены (для целей Метрики).
6. **Byte-identical модули** — что НЕ трогалось (`customer`, `admin-*`, `auth`,
   `source` — если не трогали, так и написать; помогает нам сузить review).
7. **Migration-нота консьюмеру** — какие пропсы теперь нужно передать в
   `page.tsx`/`SubmitModal`, чтобы включить новую структуру.
